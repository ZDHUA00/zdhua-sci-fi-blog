服务器最容易失控的地方不是某一个服务，而是服务之间的关系：域名指向哪里，Caddy 转发到哪个端口，Docker 容器挂载了什么目录，证书和环境变量从哪里来。

把这些关系整理成地图之后，维护动作会变得更像检查清单，而不是靠记忆操作。

## 当前维护原则

- 修改 Caddy 之前先备份配置，并确认目标域名没有被其它服务使用。
- 删除旧服务之前检查域名、反代、容器、进程和数据目录是否属于同一个目标。
- 对外入口尽量保持少而清晰，避免一个域名承载多个不相关职责。
- 静态站点优先托管到 GitHub Pages 或 Cloudflare Pages，不把服务器当作文件仓库。

## 这次博客清理

旧 Hugo 博客目录位于 `/root/zdhua-blog-new`。删除前确认它没有被 Caddy、systemd、Docker 或运行进程引用，目录内容也确实是旧博客构建产物。

删除后，博客入口迁移到 GitHub Pages。服务器只保留动态服务和反代能力，职责边界更清楚。

## 长期收益

当域名、容器和反代关系可见后，新的服务上线会更稳：先定义入口，再定义运行时，再定义数据目录，最后才是自动化部署。顺序清楚，回滚也更容易。
